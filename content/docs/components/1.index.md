Components carry the happydesigns system into product interfaces. This section documents how Nuxt UI components should feel once the brand layer is applied: structured, useful, calm, and precise.

This guide does not replace the Nuxt UI API documentation. It explains brand behavior, component hierarchy, and which parts of the styling belong in the shared theme.

## Component Philosophy

- Start with useful behavior, accessible states, and clear hierarchy.
- Use borders, spacing, and semantic surfaces before decorative color.
- Keep controls compact, predictable, and easy to scan.
- Use coral for focus, active states, and signature emphasis.
- Keep one-off component styling out of pages unless the layout itself requires it.

## Theme Ownership

Shared component styling belongs in `app/app.config.ts`. Token and semantic surface behavior belongs in `app/assets/css/main.css`.

Docs pages should compose real Nuxt UI components and explain usage. They should not become the place where component colors, sizes, variants, or slot styles are redesigned one example at a time.

## Coverage Model

The Components section uses three levels of coverage:

| Level | Meaning |
| --- | --- |
| Styled | The component has shared theme behavior in `app/app.config.ts`. |
| Showcased | The component appears in a docs example or component-family showcase. |
| Listed | The component is part of the catalog and needs brand review before deeper documentation. |

The goal is not to duplicate every Nuxt UI prop. The goal is to make the happydesigns layer consistent wherever Nuxt UI components appear.

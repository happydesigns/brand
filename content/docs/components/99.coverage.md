The happydesigns layer should eventually cover every standard Nuxt UI component available in this project. Coverage means each component has a clear role in the brand system, either through a focused example or through a larger integrated pattern. This guide does not repeat the full Nuxt UI documentation.

## Review Priority

1. Form controls and validation states.
2. Feedback states, especially warning and error behavior.
3. Tables and dense data surfaces.
4. Overlays and destructive flows.
5. Layout, docs, and prose surfaces.
6. Navigation components across docs and product contexts.
7. Product-specific groups: dashboard, publishing, chat, editor, pricing.

## Theme Checklist

- Shared visual behavior lives in `app/app.config.ts`.
- Semantic colors and surface variables live in `app/assets/css/main.css`.
- Docs examples use real Nuxt UI components.
- Page-specific classes are limited to layout, spacing, and demo composition.
- New components are assigned to a family before they get one-off styling.
